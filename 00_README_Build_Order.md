# TriagePlus — Build Guide

**Project:** TriagePlus · IIT Dharwad Summer of Innovation · Team "Hardly Human"

This repo is built from four build prompts, each a complete, standalone specification for one part of the system:

1. `01_Database_Schema_Build_Prompt.md`
2. `02_Backend_Architecture_Build_Prompt.md`
3. `03_AI_Engine_Build_Prompt.md`
4. `04_Frontend_Build_Prompt.md`

They share one contract (table names, `triage_level`/`confidence` conventions, session identity) — build them in the order above, since each later document assumes the previous ones exist.

## What's already provided vs. what gets built

**Already in this repo, not to be rebuilt:**
- `frontend/` — the landing page and chat UI (`ChatWindow`, WebSocket connection logic) from a previous build. The frontend prompt (§1) says to inventory this existing code first and extend it — add dark mode, mobile responsiveness, the mic button, and the booking chips as changes to what's there, not a rewrite.
- `backend/data/medquad/`, `backend/data/meddialog/`, `backend/data/ddxplus/`, `backend/data/symptom2disease/` — the raw source datasets for RAG and the Knowledge Graph. Nothing needs to be downloaded; the AI engine prompt's parsing/chunking code should read directly from these paths.

**Built from zero, per the prompts:**
- The full Supabase schema and RPC functions (`01`).
- The FastAPI backend: WebSocket endpoints, intake FSM, REST API, RPC integration (`02`).
- The LangGraph state machine, XGBoost classifier, Knowledge Graph loader, and the FAISS index build scripts that turn the raw datasets into `backend/faiss/` (`03`).
- The Doctor Portal and the RAG Diagnostics Monitor on the frontend; the dark mode toggle, mobile layout, and booking chips layered onto the existing landing/chat code (`04`).

## Build Order

1. **Database** — apply `supabase/migrations/0001_init.sql`, run the seed script. Nothing downstream works without this.
2. **AI Engine offline step** — before the backend can start cleanly, run `backend/scripts/build_faiss_indices.py` once against the provided raw datasets to produce `backend/faiss/medquad/` and `backend/faiss/conversations/`. Also confirm the DDXPlus loader in `backend/app/core/kg.py` parses `backend/data/ddxplus/` correctly (see AI engine prompt §12 for its exact expected format).
3. **Backend** — build the FastAPI app, wire the LangGraph nodes from the AI engine prompt into `app/core/triage_graph.py`, bring up `/ws/chat`, `/ws/diagnostics`, and the REST routes.
4. **Frontend** — inventory the existing landing/chat code, extend it per the frontend prompt, then build the Doctor Portal and Diagnostics Monitor against the now-running backend.

## Environment Variables

**`backend/.env`:**
```
SUPABASE_URL=
SUPABASE_SERVICE_ROLE_KEY=
DEVELOPER_PASSWORD=
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=llama3.2
CORS_ALLOWED_ORIGINS=http://localhost:5173
PORT=8000
```

**`frontend/.env`:**
```
VITE_API_BASE_URL=http://localhost:8000
VITE_WS_BASE_URL=ws://localhost:8000
VITE_SUPABASE_URL=
VITE_SUPABASE_ANON_KEY=
```

Note: the backend uses the Supabase **service-role** key (full access, server-side only, never shipped to the frontend); the frontend uses the **anon** key, used exclusively for Supabase Auth (doctor login), never for direct data queries — see backend prompt §7 and database prompt §4.

## Local Services Checklist

Before running the backend, have these running/available:
- **Ollama**, running locally with the `llama3.2` model pulled (`ollama pull llama3.2`) — used for LLM rewriting only, never for clinical decisions.
- **A Supabase project** (cloud or local via the Supabase CLI) with the migration from `01_Database_Schema_Build_Prompt.md` applied.
- **Python 3.10+** and **Node 18+** on the machine building the backend and frontend respectively.
